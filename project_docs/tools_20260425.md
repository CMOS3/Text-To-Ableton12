# Ableton MCP + Gemini API Tool Status (2026-04-25)

Based on a review of the backend code (`gemini_client.py`) and the Ableton remote script (`__init__.py`), here is the complete breakdown of the available tools, compound tools, their purposes, and their current working status.

There are **2 broken tools** that have implementations on the client side but are missing the corresponding endpoints in the Ableton Remote Script. The code has been left as-is without modifications.

## 🔴 Broken / Unimplemented Tools

1. **`get_track_info`**
   - **What it attempts to do:** Fetch detailed info about a specific track by its index.
   - **Status:** **BROKEN**. The Python backend sends a `"get_track_info"` request, but the Ableton remote script's `_dispatch_method` is missing this endpoint entirely. Calling it results in an "Unknown method" error.
2. **`set_track_volume_by_name` (Compound Tool)**
   - **What it attempts to do:** Find a track by name using `get_session_info`, then send a command to set its volume in dB.
   - **Status:** **BROKEN**. While the backend logic successfully finds the track index, it then tries to send a `"set_track_volume"` command to the proxy. The Ableton remote script does not have a handler for `"set_track_volume"`.

## 🟢 Fully Working Tools

### 1. System & Transport
- **`test_ableton_connection`**: Sends a `"ping"` payload to verify the Ableton Server is running and reachable.
- **`get_song_scale`**: Retrieves whether scale mode is active, the root note, and the scale name (e.g., Minor, Major).
- **`get_session_info`**: Retrieves a large payload containing all tracks, their armed states, volume, panning, current tempo, playing status, and global scale info.
- **`set_tempo`**: Changes the BPM of the session.
- **`start_playback` / `stop_playback`**: Controls the master transport in Ableton.

### 2. Track Management
- **`create_midi_track`**: Creates a new MIDI track at the end of the session and names it.
- **`delete_track`**: Deletes a track using its integer index.
- **`set_track_name`**: Renames an existing track based on its index.
- **`select_track`**: Selects/focuses a track in the Ableton UI by its name.
- **`arm_track`**: Arms or disarms a specific track for recording by name.

### 3. Clip & MIDI Operations
- **`create_clip`**: Creates an empty MIDI clip of a specified length (in beats) on a target track and slot index.
- **`delete_clip`**: Deletes a clip from a specific track and slot index.
- **`set_clip_name`**: Renames an existing clip.
- **`fire_clip` / `stop_clip`**: Triggers playback or stops a specific clip slot.
- **`add_notes_to_clip`**: Takes an array of notes using semantic pitch names (like 'C3', 'Eb2'), translates them to MIDI integers locally, and injects them into the clip.
- **`get_notes_from_clip`**: Returns an array of all notes currently inside a specific clip.
- **`delete_notes_from_clip`**: Removes specific notes from a clip by precisely matching both pitch and start time.
- **`inject_midi_to_new_clip`**: Automates finding the first empty clip slot on a track, creating a clip of a given length, and injecting notes in one single transaction.

### 4. Browser & Devices
- **`get_browser_tree`**: Retrieves the root-level folders of the Ableton browser (Sounds, Drums, Instruments, Audio Effects, etc.).
- **`get_browser_items_at_path`**: Browses deeper into the folder tree (e.g., `"Packs/Lost and Found"`) and returns its children files/folders.
- **`load_instrument_or_effect`**: Loads a device from the browser onto a track.
- **`load_drum_kit`**: Explicitly loads a drum kit from the browser onto a track.
- **`get_track_devices`**: Lists the names and indexes of all devices loaded on a specific track.
- **`get_device_parameters`**: Retrieves all parameters for a device, including their names, current values, minimums, and maximums.
- **`set_device_parameter`**: Sets a parameter's value. It includes a robust fuzzy-matching algorithm and synonym map (e.g., understanding that "cutoff", "freq", and "filterfreq" are the same thing) to find the correct parameter even if the AI slightly misspells it.

### 5. Working Compound / Special Tools
- **`get_session_mix_status` (Compound Tool)**: This tool runs entirely on the Python backend side. It calls `get_session_info`, parses the result, and formats a highly readable, multi-line string showing the volume and panning for every track.
- **`ui_text_response` (Synthetic Tool)**: This is a fake tool defined in the schemas but handled directly by the frontend stream. It's used to route conversational text, markdown formatting, or advice back to the chat UI instead of Ableton.
